# Week 2 Notes

- Day 1:
  - 做了什么：运行 `./bench` 对 `N=512/1024/2048` 做 row-major vs col-major 基线对比。
  - 数据摘要：
    - `N=512`: row `1.17 ns/elem`，col `1.14 ns/elem`（差异很小）
    - `N=1024`: row `1.07 ns/elem`，col `1.06 ns/elem`（差异很小）
    - `N=2048`: row `0.94 ns/elem`，col `2.17 ns/elem`（col 明显变慢，约 2.3x）
  - 解释：矩阵变大后，col-major 访问步长为 `N`，空间局部性变差，cache line 利用率下降；row-major 连续访问，局部性更好。
  - 结论：当数据规模超过缓存友好范围时，访问顺序会成为主导性能因素。
  - 明日计划：进入 Day 2，使用 `cache_sim` 跑 trace，验证 hit/miss/eviction 与访问模式之间的关系。
- Day 2:
  - 做了什么：实现并运行 `cache_sim`，重点验证地址拆分、`M` 双访问语义、以及 `E` 对冲突 miss 的影响。
  - 数据摘要：
    - `./cache_sim -s 2 -E 1 -b 3 -t traces/sample.trace` -> `hit=4 miss=8 eviction=4`
    - `./cache_sim -s 2 -E 2 -b 3 -t traces/sample.trace` -> `hit=5 miss=7 eviction=0`
  - 解释：
    - `E=1` 时同一 set 只能放 1 条 line，冲突更容易触发 eviction。
    - `E=2` 时同一 set 可并存 2 个 tag，本样例中冲突明显缓和，eviction 降为 0。
    - `M` 被模拟为两次访问（读+写），常见模式是第一次 miss、第二次 hit。
  - 结论：`s/b` 决定地址如何映射到 set 和 block，`E` 决定同一 set 的容纳能力；三者共同决定 hit/miss/eviction。
  - 明日计划：进入 Day 3，跑 baseline transpose，聚焦“读连续、写跨行”的访存瓶颈。
- Day 3:
  - 做了什么：运行 `./transpose_baseline` 的多规模实验（`N=256/512/1024`），建立无 blocking 的转置趋势基线。
  - 数据摘要：
    - `N=256`: `total=0.652 ms`, `avg=0.130 ms/iter`, `1.99 ns/elem`, `checksum=3144630.0`
    - `N=512`: `total=3.507 ms`, `avg=0.701 ms/iter`, `2.68 ns/elem`, `checksum=12581737.0`
    - `N=1024`: `total=20.215 ms`, `avg=4.043 ms/iter`, `3.86 ns/elem`, `checksum=50331376.0`
  - 解释：
    - 朴素转置 `B[j][i] = A[i][j]` 内层 `j` 递增时，读取 `A` 连续，读路径局部性较好。
    - 写入 `B[j][i]` 则是按列跳写，地址步长约为 `N * sizeof(float)`（`N=1024` 时约 `4096B`），空间局部性差。
    - 在常见 `write-back + write-allocate` 策略下，写 miss 会触发 RFO（把目标 line 先读入 cache），随后再写脏并在未来回写；若写入分散，会放大内存流量成本。
    - 若按 `64B cache line`、`float=4B` 粗估，一条 line 可容纳 `16` 个 float；朴素写法常出现“每次只用 line 中 1 个 float”的低利用率现象。
    - 从 `1.99 -> 2.68 -> 3.86 ns/elem` 的增长可见：矩阵变大后，cache 友好度下降，单位元素代价上升。
  - 结论：baseline 的主要瓶颈在 `B` 的写路径（跨步写 + 低 line 利用率），而非算术计算本身；且这种瓶颈会随 `N` 增大而更明显。
  - 明日计划：进入 Day 4，加入 blocking，并测试不同 `BS` 的性能差异。
- Day 4:
  - 做了什么：运行 `./transpose_blocking 1024 5 BS` 扫描不同块大小，比较 naive 与 blocked。
  - 数据摘要（`N=1024,iters=5`）：
    - `BS=4`: blocked `3.958 ms`，`5.34x`
    - `BS=8`: blocked `1.664 ms`，`10.93x`
    - `BS=16`: blocked `1.644 ms`，`10.72x`
    - `BS=24`: blocked `1.775 ms`，`9.83x`
    - `BS=32`: blocked `2.027 ms`，`8.64x`
    - `BS=48`: blocked `8.269 ms`，`2.10x`
    - `BS=64`: blocked `8.726 ms`，`2.00x`
    - `BS=96`: blocked `9.345 ms`，`1.84x`
    - `BS=128`: blocked `12.834 ms`，`1.34x`
  - 解释：
    - `BS` 过小：块切分过细，循环控制开销比例上升，收益被部分抵消。
    - `BS` 适中（本机约 `8~24`）：工作集更容易留在 L1/L2，A/B 重用距离缩短，性能最佳。
    - `BS` 过大（`>=48`）：单块数据超出 cache 友好范围，冲突与替换增加，性能明显回落。
  - 结论：blocking 显著有效（远超验收建议的 `20%`），但最优 `BS` 强依赖硬件与编译环境，不能盲目照搬。
  - 明日计划：进入 Day 5，整理复盘并沉淀优化模板。
- Day 5:
  - 本周复盘（性能坑）：
    - 坑 1：只看算法复杂度，不看访存路径；同样 `O(N^2)`，性能可相差一个数量级。
    - 坑 2：误以为“读重要、写不重要”；实际写路径同样受 cache 策略与局部性强烈影响。
    - 坑 3：固定一个 `BS` 到处用；在不同机器/参数下可能从最优变次优。
  - 本周复盘（有效策略）：
    - 策略 1：先做 baseline，再做单变量优化（只改一个因素，便于归因）。
    - 策略 2：优先优化访问顺序与重用距离（让热数据更快被再次访问）。
    - 策略 3：参数扫描 + checksum 校验（保证“更快”同时“结果正确”）。
  - 下周风险（1 条）：
    - 风险：过拟合当前机器缓存特性，迁移到其他 CPU 后性能结论不稳定。
  - 访问模式优化模板（可复用）：
    - 步骤 1：画出读/写地址序列（谁连续、谁跨步）。
    - 步骤 2：估算步长与 line 利用率（例如 `stride / line_size` 直觉判断）。
    - 步骤 3：先改循环顺序，再尝试 blocking。
    - 步骤 4：扫描关键参数（如 `BS`），记录趋势而非只看单点。
    - 步骤 5：每次实验都保留正确性校验（checksum）与环境信息。
